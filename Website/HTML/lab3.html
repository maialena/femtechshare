WEEK#3: HTML Syntax & Elements
OVERVIEW
HTML stands for Hyper Text Markup Language. It's what you use to create the bare-bones layout of your website. Think of HTML as a plain ole' Word Document with just the content, and CSS as the styling (font, color, bold, size, etc.).
LEARNING OBJECTIVES
  1. Understand the basic building blocks of an HTML5 Doc
  2. Feel comfortable cloning a website in HTML
  3. Add style to your page with CSS

Lesson 1: Tags, Attributes, & Semantic Markup

  1. Tags are like containers -- they 'hold' information. They are always enclosed in < > brackets. Most tags include a starting and closing tag like so: 
     This is a line of text
    Notice in this example that the closing tag has a forward slash ('\').

    Here's a list of the most common tags in HTML5

  2. Attributes tell us more about the elements. Most attributes can be used only on certain elements, but a few (like "lang") can be used on any elements. 
     <p lang="en-US" > Paragraph in English </p>
  3. Semantic Markup refers to HTML text elements that are not used for markup, but instead add some style to the page -- like bolding, underlining, and italicizing (to name a few).

Pro Tip: It's best practice to do as LITTLE styling in your HTML file, since styling is mainly CSS territory. You may think that adding an <em> </> tag to your text isn't a big deal. And you're right -- it's okay to use sparingly. But what if down the line you wanted to change the italic font to bold...then you'd have to sift through your entire html file to find all the times you used the emphasis tags. Wouldn't it be easier to use the <span> tag instead? That way you can just go to your SINGLE style.css file and change one line of code, instead of (potentially) hundreds....(we'll learn more about CSS soon).

Pro Tip: Pair programming is often harder than it sounds. It's all about developing the right workflow to allow both people to contribute. WHat you don't want to happen is have one person type, while the other is on her phone playing Pokemon Go. So here's one suggestion: tradeoff typing. While Person A is at the keyboard, Person B should still follow along and dictate lines of code for Person A to type in. After some time, switch.
